# JobQueue Sample

An async job processing system demonstrating **step functions** and **server-generated values**.

## What It Demonstrates

### Pattern 1: Step Functions (Observing Async State Changes)
- `CreateJob` returns immediately with `Pending` status
- Background processing eventually completes (or fails)
- `GetJob` uses `Expect.OneOf` — response could show `Pending` or `Completed`
- Whichever we observe, the model updates accordingly

### Pattern 2: Server-Generated Values (Learning from Response)
- `ResultPath` is generated by the server when job completes
- While `Pending`: `ResultPath` is `null` (unknown)
- Once `Completed`: `ResultPath` is revealed and captured in state
- This is **response-dependent state** — we learn the value from the response

### Temporal Properties
- Status transitions: `Pending` → `Completed` or `Failed`
- `ResultPath` stability: once set, never changes
- Polling with derivations from prior responses

## Key Patterns

### Pattern 1: Step Function (Multiple Possible Outcomes)

```csharp
// GetJob can observe EITHER "still pending" OR "now completed"
if (job.Status == JobStatus.Pending)
{
    return Expect.OneOf(
        // Outcome 1: Still pending - no state change
        Expect.That<ApiResult<Job>>(r => r.Data.Status == JobStatus.Pending)
            .SameState(),
        
        // Outcome 2: Now completed - transition state
        Expect.That<ApiResult<Job>>(r => r.Data.Status == JobStatus.Completed)
            .ThenState((response, next) => {
                next.Jobs[jobId].Status = JobStatus.Completed;
                // ...
            })
    );
}
```

### Pattern 2: Server-Generated Value (Learn from Response)

```csharp
// ResultPath is generated by server - we learn it from the response
.ThenState((response, next) => {
    next.Jobs[jobId].ResultPath = response.Data.ResultPath;  // Server-generated!
}, mock: () => new ApiResult<Job> {
    Data = new Job { ResultPath = "/results/abc123" }  // Mock for exploration
});
```
